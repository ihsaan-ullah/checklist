Question # 5: Does the paper provide open access to the data and code, with sufficient instructions to faithfully reproduce the main experimental results, as described in supplemental material?

Author's answer: Yes

Author's justification: Our code is made available in the supplementary materials, and links to the datasets can be found as references. In this version, to protect anonymization, we are not publishing the code on our GitHub repo, but this will be done for the final version.

LLM Review:
Feedback:
  1. Experimental Validation and Reproducibility: The authors claim that their code and datasets are provided in the supplementary materials, which is crucial for experimental reproducibility. However, there is no explicit mention or evidence in the main text about the "exact command and environment needed to run to reproduce the results," which falls short of the NeurIPS guidelines for ensuring reproducibility. It is essential that the authors include a clear and detailed explanation or script that demonstrates how to replicate their experimental setup and results, including environment specifics (e.g., software versions, hardware requirements).

  2. Clarity on Dataset Access and Preprocessing Instructions: While the paper mentions that links to the datasets can be found as references, and datasets are ostensibly provided in the supplementary materials, the exact process for accessing, preparing, and using the datasets to reproduce results lacks detail. Clear instructions on how to access and preprocess the data are necessary. Additionally, specifying any licensing information or restrictions for the dataset is crucial to ensuring open access and usability by the broader research community.

  3. Code Availability for Anonymity: The authors addressed the anonymization appropriately by stating that the GitHub repository would be published in the final version, adhering to NeurIPS guidelines for submission anonymity. For the final version, ensuring the code repository is well-documented, including README files with detailed setup, execution instructions, and specifying any dependencies, will be critical for fostering reproducibility and adoption by the community.

  4. Supplementary Material Detailing: It is recommended that the authors include a more detailed description of the supplementary material in the main text, guiding readers on what exactly is provided and how it complements the paper. For instance, specifying whether the supplementary material includes additional results, theoretical proofs, detailed architecture diagrams, or datasets would be beneficial.

  5. Baseline Comparisons and Error Bars: Assuming error bars were omitted due to space constraints or computational limitations, it would be beneficial for the authors to clarify this choice explicitly. For future work or in supplementary materials, providing error bars or confidence intervals around reported metrics could enhance the credibility of the experimental results, especially if the results demonstrate significant improvements over existing methods.

  6. Licenses of Datasets: The concern regarding dataset licenses was appropriately anticipated in the guidelines. If not already included, adding a statement about each dataset's license used in the experiments, or stating that all datasets are publicly available and open for research use, would resolve any ambiguity around dataset permissions and reuse.

Concluding Recommendations:
  • Explicitly detail the environment setup and exact commands for reproducing experiments in the supplementary material or an appendix within the paper.
  • Provide comprehensive instructions for accessing, preprocessing, and utilizing the datasets, including any necessary scripts.
  • In the final version, ensure the code repository is detailed, well-documented, and publicly accessible, including clear licensing information.
  • Enhance the description of supplementary materials within the main text to clearly indicate its contents and relevance to the paper's contributions.

The paper needs improvements on providing clear, accessible reproduction instructions and dataset information to align with the NeurIPS guidelines fully, although the issue of code availability due to anonymization is handled correctly.

Question # 8: For each experiment, does the paper provide sufficient information on the computer resources (type of compute workers, memory, time of execution) needed to reproduce the experiments?

Author's answer: Yes

Author's justification: Compute Resources are explained in the "Experimental conditions" section.

LLM Review:
Feedback on the provided “Answer” and “Justification” for the NeurIPS checklist question regarding disclosure of compute resources:
  1. Room for Improvement in Experimental Conditions Disclosure: The paper mentions that the compute resources are explained in the "Experimental conditions" section, but it does not provide specific details about the type of compute workers (CPU/GPU), internal cluster, or cloud provider. Additionally, the authors do not mention relevant memory, storage, or the amount of compute (in hours or GPU/CPU time) required for each of the individual experimental runs. This lack of detailed breakdown limits the reproducibility of the experiments. The authors should include a table or a detailed paragraph listing the exact specifications of the hardware used (model of GPU/CPU, amount of RAM, etc.), execution time for each experiment, and an estimate of the total compute resources required. If any experiments required more compute than reported, this disparity should also be disclosed.

  2. Clarity and Detailing in Compute Resources Description: In the experimental validation section (Section 5), while the authors provide some information about the training duration (e.g., "Training is done for 280,000 iterations with batch size 100, and lasted 48h."), there is no explicit mention of the hardware used. This is crucial information for reproducing the experiments and understanding the computational efficiency of the proposed Deep Statistical Solvers (DSS). Authors should clarify whether GPUs or CPUs were used, the specifics of these units, and if any special computational techniques (e.g., distributed computing) were employed.

  3. Potential Inconsistencies in Reporting Compute for Different Experiments: The paper discusses two main experimental validations - solving linear systems and AC power flow experiments. However, the clarity on compute resources appears to vary between these sections. For example, the linear systems experiment mentions the duration, but similar detailed information in the AC power flow experiment is lacking specificity about compute resources. Consistency should be maintained across sections in reporting the compute resources.

  4. Omission of Preliminary or Failed Experiments Compute: According to the guidelines, if the full research project required more compute than the experiments reported in the paper, this should be disclosed. This includes preliminary or failed experiments that did not make it into the final paper. Such disclosure is essential for a full understanding of the computational cost of developing the presented methods. The paper should add a paragraph discussing any such experiments, even if briefly, to comply with this guideline.

  5. Explicit Mention of Internal Cluster or Cloud Provider: If the experiments were conducted on an internal cluster or a cloud provider, specifying this information would be helpful. These details can critically impact the reproducibility and feasibility of the experiments for other researchers, especially those with limited access to computational resources.

Actionable Feedback:
  • Include a detailed breakdown of the compute resources for each experimental run, specifying the type (CPU/GPU), model, memory, storage, and the execution time.
  • Ensure consistency in detailing the compute resources across all experimental validations presented in the paper.
  • Discuss any significant computational efforts beyond those reported, including preliminary or exploratory experiments, to give a complete view of the computational investment.
  • Specify whether experiments were run on an internal cluster or cloud provider, including any relevant details such as the provider name or configuration specifics.
